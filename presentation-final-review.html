1 What this project is

CareerX is a web application that turns a student’s questionnaire answers into a rich, personalized career dashboard: pathways, universities, certifications, jobs with portal links, a five-year skill roadmap, optional scholarships or startup resources, and citations to real URLs gathered from the live internet.

Unlike a quiz that only outputs static templates, this system retrieves current evidence from search results and feeds that text into a large language model (LLM), then forces the model to answer in a strict JSON shape. That design is the core of Retrieval-Augmented Generation (RAG).

2 Main algorithm: Retrieval-Augmented Generation (RAG)

RAG means: first retrieve relevant documents from an external source, then generate an answer that is grounded in those documents. Pure LLMs can hallucinate facts; RAG reduces that by tying outputs to retrieved text.

In this codebase, RAG is implemented end-to-end in app/api/generate-dashboard/route.ts:

  1. Plan (LLM) — Gemini generates profileSummary plus 6–8 diverse search queries (Reddit, LinkedIn, salaries, local institutions, next-degree programs, etc.). Current year is injected server-side so the model does not invent outdated years.
  2. Retrieve (SearXNG) — Each query runs against self-hosted SearXNG with up to 5 pages per query; results are merged, deduplicated by URL, and filtered (e.g. dictionary sites blocked).
  3. Deep retrieval (scraper) — The top 10 URLs are fetched and full page text is extracted (“deep RAG”). Many more results stay as snippets only for breadth.
  4. Generate (LLM + schema) — All retrieved material is passed into Gemini with strict instructions (currency, location, education-level rules, citation rules). Output must match DashboardDataSchema via generateObject.

Safeguard: If fewer than 5 unique results are returned, the API responds with an error instead of fabricating a dashboard — explicitly to avoid hallucination when search is down or rate-limited.

4 Career pathway scoring (explainable heuristic)

The synthesis prompt defines a weighted alignment formula so match percentages are not arbitrary prose:

Score = 0.4 × Skill_Match + 0.3 × Market_Demand + 0.3 × User_Preference
  • Skill_Match (0–100) — Alignment between the user’s background and pathway requirements.
  • Market_Demand (0–100) — Inferred from retrieved labor-market and trend signals in the research text.
  • User_Preference (0–100) — Fit to questionnaire answers, location, and stated goals.

This is a transparent rubric for reviewers: scores are guided by explicit weights even though individual sub-scores are estimated by the model from retrieved context.

5 What is Zod? Why use it?

Zod is a TypeScript-first schema library: you describe the shape and rules of data (strings, numbers, enums, nested objects, arrays, min/max lengths, etc.). At runtime it can validate unknown JSON; at compile time z.infer gives you accurate TypeScript types.

Why Zod in CareerX?

  • Structured LLM outputs — The Vercel AI SDK’s generateObject uses your Zod schema (via JSON Schema conversion) so Gemini returns data that fits DashboardDataSchema: careers, universities, jobs, roadmaps, sources, optional persona blocks.
  • Fewer broken UIs — The frontend expects consistent fields (e.g. every job can have jobPortalLinks). Validated objects prevent random missing keys from crashing components.
  • Single source of truth — Types and validation live together in lib/schemas.ts (questionnaire types, dashboard types, ATS scan schema in scan-resume).

In short: Zod turns “free-form AI text” into reliable, typed JSON your app can render safely.

6 What is SearXNG? Why not Google-only?

SearXNG is an open-source metasearch engine: one query can aggregate results from multiple backends (configurable), without locking you into a single vendor’s API.

How this project uses it

  • HTTP POST to /search with form fields for query, language, pagination (pageno).
  • HTML responses are parsed with Cheerio to extract titles, URLs, and snippets.
  • In-memory cache (~6 hours) avoids hammering the instance for repeated queries.
  • Staggered queries and delays between pages reduce burst rate limits.

Self-hosting matters for demos and research: you control rate limits, privacy, and availability instead of depending on a fragile public instance.

7 Docker, containers, and why they appear in this project

Containerization packages an application plus its dependencies into an isolated, repeatable unit called a container. It runs consistently on any machine that has a container runtime (e.g. Docker).

What is Docker?

Docker is a platform to build, ship, and run containers. Docker Compose (your docker-compose.yml) declares services, ports, volumes, and environment variables so one command starts SearXNG the same way every time.

Why Docker for SearXNG here?

  • Fast setup — docker compose up -d pulls searxng/searxng:latest and exposes port 8080.
  • Configurable instance — Local folder ./searxng is mounted to /etc/searxng for settings.
  • Limiter disabled for development — SEARXNG_DISABLE_LIMITER=1 supports heavier automated querying during research (document trade-offs for production).
  • Hardening hints — Compose drops Linux capabilities (cap_drop: ALL) and only adds what is needed — good talking point for security-aware reviewers.

9 Other features reviewers often ask about

  • Stateless / privacy angle — Session data can live client-side (sessionStorage) without a central user database; scraping uses timeouts and rotating User-Agents (see README).
  • Chat endpoint — app/api/chat/route.ts streams replies grounded in the generated dashboard JSON context.
  • Resume / ATS scan — Separate API uses Zod (AtsResultSchema) for match score, missing keywords, and short feedback.
  • Evaluation / ablation — Scripts compare LLM-only vs full RAG pipeline; JSON results in repo document methodology.
